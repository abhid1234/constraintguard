# PRODUCT — `cg extract`: pull declared constraints out of a context

Issue: [#2](https://github.com/abhid1234/constraintguard/issues/2) · Depends on: #1 (constraint schema + validator)

## Problem / motivation

Agents declare their rules inside free-form context — a system prompt, a rules
section, a task brief. Before those rules can be pinned back after compaction or
scored for survival, ConstraintGuard has to lift them out of that prose into the
open constraint set schema (from #1). `cg extract <context-file>` is that first
step: read a text/markdown context and emit a well-formed constraint set. The
one real decision — *what marks a "declared constraint" in free-form text* — is
resolved below, because the marker choice shapes the schema-facing contract.

## Recommended direction

**An explicit fenced constraint block is the marker.** The author wraps their
constraints in a fenced code block whose info string is `constraints`:

~~~markdown
```constraints
must [no-pii]: Never include personal data in output.
should: Prefer the shortest correct answer.
must: Refuse tasks outside the approved scope.
```
~~~

Each non-empty line inside the block is one constraint. This is explicit,
harness-neutral (a plain markdown fence any tool renders), deterministic to
parse with the Node standard library, and has a near-zero false-positive rate —
nothing outside a `constraints` fence is ever treated as a rule. It maps
cleanly onto `{ id, text, severity }` without guessing at intent.

Rejected alternatives (see TECH.md for detail): heading convention (`##
Constraints` + list) — ergonomic but high false-positive and formatting-fragile;
inline markers (`@constraint …`) — flexible but noisy and easy to trip
accidentally; NLP/LLM inference from prose — violates zero-dependency and
determinism, and is not reproducible.

## Line grammar (inside a `constraints` block)

```
<severity> [<optional-id>]: <text>
```

- **severity** — `must` or `should` (case-insensitive), required, at line start.
  Maps directly to the schema's `severity` field.
- **[id]** — optional, in square brackets. If present it becomes the
  constraint's `id` verbatim. If absent, a stable id is generated by slugifying
  the text (see TECH.md), so ids are deterministic and human-readable.
- **text** — everything after the colon, trimmed; must be non-empty.
- Blank lines and lines beginning with `#` (comments) are ignored.

## Desired behavior

- `cg extract session.md` reads the file, finds every `constraints` fenced
  block (in document order), parses each line, and prints the resulting
  constraint set as pretty JSON to **stdout** — so it pipes straight into the
  other commands: `cg extract session.md | cg validate -` (future) or
  `cg extract session.md > constraints.json`.
- The emitted set always passes `validateConstraintSet` from #1 (unique ids,
  valid severities, non-empty text).
- **No constraints found** is a normal result, not an error: print `[]` and exit
  `0`, with a one-line note to **stderr**.
- **Malformed lines** (missing severity, missing colon, empty text) are skipped
  by default; each skipped line is reported to stderr with its line number, and
  extraction continues. `--strict` turns the first malformed line into a
  non-zero exit with a clear error instead.
- **Duplicates** are removed: two lines that resolve to the same
  `(id, text, severity)` collapse to one. Two lines that claim the *same explicit
  id* but differ in text/severity are an id conflict — keep the first, warn on
  stderr (or, under `--strict`, error).
- The library exposes the same logic as a pure function for programmatic use.

## Acceptance criteria

- [ ] `extractConstraints(text)` (library) returns an array of
      `{ id, text, severity }` for every valid line across all `constraints`
      fenced blocks, in document order.
- [ ] Output always validates against `validateConstraintSet` (#1): severities
      are `must`/`should`, ids are unique, text is non-empty.
- [ ] `must`/`should` are parsed case-insensitively into `severity`.
- [ ] An explicit `[id]` is used verbatim; a missing id is filled by a
      deterministic slug of the text, disambiguated on collision.
- [ ] Multiple `constraints` blocks in one file are all extracted, in order.
- [ ] No `constraints` block → returns `[]`; the CLI prints `[]`, exits `0`, and
      notes it on stderr.
- [ ] Malformed lines are skipped with a stderr warning naming the line number;
      valid lines from the same block are still extracted.
- [ ] Identical constraints are de-duplicated; conflicting explicit ids are
      reported and the first kept.
- [ ] `--strict` makes malformed lines and id conflicts a non-zero exit.
- [ ] `cg extract <file>` prints the pretty-printed JSON set to stdout; a
      missing/unreadable file path exits non-zero with a clear message.
- [ ] Zero runtime dependencies; the parser is pure and deterministic.
- [ ] `npm test` covers happy path, generated ids, multi-block, no-block,
      malformed-line, dedup, and CLI stdout.

## Non-goals

- Inferring constraints from ordinary prose (no NLP/LLM, no heuristics beyond the
  explicit block). Heading and inline-marker syntaxes are not supported in v1.
- Writing or modifying context files (extract only reads).
- `pin`, `conformance`, or scoring — separate roadmap items.
- Any schema change to #1. Extract emits the existing shape; it does not extend
  it (e.g. no source-location or provenance fields yet).

## Open questions (for the reviewer)

1. **Marker syntax is the human gate for this issue.** The spec recommends the
   explicit `constraints` fenced block. Confirm this over the heading/inline
   alternatives before implementation, since it fixes the authoring contract.
2. **Generated id style** — slug-of-text (readable, stable per line;
   recommended) vs. sequential `c1..cN` (simpler, but shifts when lines are
   inserted). Spec assumes slug; flag if you prefer sequential.
